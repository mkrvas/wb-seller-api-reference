# Дизайн: автообновление справочника из официальных OpenAPI-спек WB

Дата: 2026-07-22
Статус: одобрен, ждёт ревью финального текста

## Контекст

Репозиторий `mkrvas/wb-seller-api-reference` — справочник по Wildberries Seller API
в формате Claude Code Skill (SKILL.md + references/*.md). Опубликован 2026-07-21.

Слабое место: справочник обновляется руками и со временем отстаёт от реальных
изменений WB API. Цель этого проекта — добавить конвейер автообновления из
официальных OpenAPI-спецификаций `dev.wildberries.ru`, сохранив главную ценность
справочника: рукописные знания (проверенные лимиты, стратегии ретраев, грабли,
даты deprecation), которых в спеках нет и не будет.

## Цели

1. Таблицы эндпоинтов в references/*.md генерируются из официальных спек и
   обновляются автоматически (GitHub Actions, ежедневно).
2. Рукописные знания физически защищены от перезаписи генератором.
3. Изменения приходят как PR с человеческим описанием диффа — мержит человек.
4. Покрытие расширяется до всех 14 категорий официальных спек (сейчас в
   справочнике нет: заказы DBW, заказы DBS, самовывоз in-store-pickup, WBD).

## Не-цели

- Генерация SDK-кода (это справочник, не библиотека).
- Автомерж PR без человека.
- Английская версия.
- Публикация в PyPI/маркетплейсы.

### Две зоны в md-файлах

Авто-зона — таблицы эндпоинтов, оборачиваются маркерами:

```markdown
<!-- AUTO:BEGIN spec=11-analytics section=endpoints -->
| Метод | Путь | Назначение |
|---|---|---|
| POST | /api/analytics/v3/sales-funnel/products | Воронка по товарам |
<!-- AUTO:END -->
```

Правила:
- Генератор заменяет содержимое строго между `AUTO:BEGIN` и `AUTO:END`.
  Всё вне маркеров не читается и не пишется.
- Одному файлу справочника может принадлежать несколько авто-зон
  (например, эндпоинты из разных спек).
- Рукописные аннотации к конкретным эндпоинтам живут вне зон — под таблицей
  (примечания, лимиты, грабли). При исчезновении эндпоинта из спеки строка
  уходит из таблицы, а рукописное примечание остаётся — его снимает человек
  при ревью PR (это осознанно: автоудаление знаний опаснее мусорной строки).

### Что входит в авто-зону (v1)

Из каждой спеки: метод, путь, краткое назначение (summary), признак
deprecated. НЕ входят: rate limits (в спеках их нет — остаются рукописными),
модели полей ответов (объём взорвёт контекст скилла; возможное расширение v2).

Распределение эндпоинтов одной спеки по нескольким файлам задаётся в
mapping.yaml фильтрами по префиксу пути.

### Скачивание спек (fetch_specs.py)

Источник — только официальный: `https://dev.wildberries.ru/api/swagger/yaml/ru/{имя}.yaml?region=ru`,
14 имён категорий захардкожены в mapping.yaml.

Портал прикрыт анти-бот защитой, поэтому две ступени:
1. HTTP-запрос с браузерными заголовками (User-Agent Chrome, Accept и т.д.).
2. Фолбэк: headless-браузер Playwright (устанавливается только в CI).

Валидация каждого файла: ответ начинается с `openapi:` и парсится как YAML.
Политика all-or-nothing: если хотя бы одна из 14 спек не скачалась/невалидна —
скрипт завершается с ошибкой, `specs/` не трогается вообще. Это исключает
полуобновлённое состояние.

### Ночной цикл (update.yml)

Триггеры: `schedule: cron '30 3 * * *'` (03:30 UTC ежедневно) + `workflow_dispatch`
(ручной запуск кнопкой). Шаги:

1. checkout; setup python 3.12; pip install pyyaml requests; playwright ставится
   только если первая ступень скачивания не прошла.
2. `fetch_specs.py` → свежие YAML во временную папку → при успехе замена `specs/`.
3. `generate_tables.py` → перегенерация авто-зон в references/*.md.
4. `git diff` пустой → конец, до завтра.
5. Дифф есть → `check_drift.py` формирует описание (новые эндпоинты /
   изменённые / удалённые / deprecated) → ветка `auto/spec-update-<дата>` →
   PR в master через встроенный GITHUB_TOKEN (права workflow: contents+pull-requests
   write). Никаких сторонних секретов не требуется.
6. Скачивание провалилось → workflow падает красным (придёт уведомление GitHub),
   справочник остаётся вчерашним. Ничего не ломается.

### Локальное обновление скилла

Папка `~/.claude/skills/wb-api` — клон этого репозитория (уже так).
После мержа PR: `git pull` локально. Claude подтягивает свежак сам при работе
с WB-задачами; инструкция для людей — в README.

## Критерии приёмки

1. `python3 -m py_compile` проходит для всех скриптов; mapping.yaml валиден.
2. `fetch_specs.py` локально скачивает все 14 спек с официального портала.
3. `generate_tables.py` на реальных спеках выдаёт таблицы, по смыслу
   совпадающие с текущим справочником (ручная сверка минимум трёх семейств,
   включая analytics — воронку).
4. Идемпотентность: повторный прогон генератора не даёт диффа.
5. Рукописный текст вне маркеров побайтово не изменяется генератором
   (проверка диффом на файлах с обеими зонами).
6. Ручной запуск workflow в GitHub реально создаёт PR с осмысленным описанием.
7. В репозитории нет секретов/токенов (скан перед пушем) и нет упоминаний
   сторонних SDK/зеркал.

## Риски

- Анти-бот защита dev.wildberries.ru может измениться так, что и Playwright
  не пройдёт из датацентровых IP GitHub Actions. Тогда обновления встанут до
  ручной починки fetch-скрипта; сам справочник при этом жив. Обнаруживается
  по красным workflow-запускам.
- Снапшоты спек WB публикуются в открытом репо. WB раздаёт их открыто всем
  разработчикам, но формально это документы WB — при претензии убрать `specs/`
  из репо (генерация продолжит работать, спеки станут артефактом CI).
- WB может добавить 15-ю категорию спек — список имён фиксированный, новую
  надо добавлять руками в mapping.yaml. Обнаруживается при ручных сверках
  с порталом.
