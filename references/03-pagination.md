# Пагинация в WB API

У WB разные методы пагинации в зависимости от семейства API. Перед интеграцией проверь, какой метод использует конкретный эндпоинт.

## 1. Cursor-based (курсор)

**Где:** Content API (`/content/v2/cards/cursor/list`)

Первый запрос — пустой курсор, следующий — курсор из ответа.

```json
// Запрос 1
{ "limit": 100, "cursor": "" }

// Ответ
{
  "data": {
    "listGoods": [...],
    "cursor": "eyJkYXRhIjoibmV4dF9jdXJzb3IifQ=="
  }
}

// Запрос 2
{ "limit": 100, "cursor": "eyJkYXRhIjoibmV4dF9jdXJzb3IifQ==" }
```

**Стоп-условие:** `listGoods` пустой или меньше `limit` записей.

## 2. rrdid (ID строки отчёта)

**Где:** Statistics API (`/api/v5/supplier/reportDetailByPeriod`) — **финансовый отчёт**

Каждая строка отчёта имеет `rrd_id`. Следующий запрос начинается с последнего `rrd_id` из предыдущего ответа.

```
GET /api/v5/supplier/reportDetailByPeriod
    ?dateFrom=2024-06-01
    &dateTo=2024-06-30
    &rrdid=0           ← первый запрос: 0
    &limit=100000      ← максимальный лимит
```

```
// Ответ: массив строк с полем rrd_id
{ "data": [ { "rrd_id": 1001, ... }, ..., { "rrd_id": 5000, ... } ] }
```

```
// Следующий запрос
GET ...&rrdid=5000&limit=100000
```

**Стоп-условие:** Получили меньше `limit` записей.

**Лимит:** 1 запрос в минуту — строго выдерживать паузу 60 сек.

**Подводные камни:**
- `rrd_id` может быть не монотонным — при пересчёте отчёта появляются дыры
- Данные за прошлые даты могут измениться при пересчёте (не финальны < 7 дней)
- Данные доступны с 29 января 2024 года
- Рекомендуемый `limit`: 100000 (минимизирует число запросов)

## 3. Offset / Limit (смещение)

**Где:** Documents, некоторые отчёты, Feedbacks, Questions

```
GET /api/v1/documents/list?limit=100&offset=0
GET /api/v1/documents/list?limit=100&offset=100
GET /api/v1/documents/list?limit=100&offset=200
```

**Стоп-условие:** Пустой ответ или записей меньше `limit`.

## 4. Date-based (по датам)

**Где:** Statistics API (orders, sales), Advertising (history)

Запрашиваешь данные с `dateFrom` и `dateTo`. Для больших периодов — разбивай на недели/месяцы.

```
GET /api/v1/supplier/sales?dateFrom=2024-06-01&dateTo=2024-06-07
GET /api/v1/supplier/sales?dateFrom=2024-06-08&dateTo=2024-06-14
```

## 5. Для остатков (stocks)

**Где:** Statistics API (`/api/v1/supplier/stocks`)

> ⚠️ **Требует ручной проверки** (аудит от 2026-07-21): по одному источнику этот эндпоинт мог быть
> заменён 23.06.2026 на `POST /api/analytics/v1/stocks-report/wb-warehouses` (Analytics API, offset-пагинация,
> лимит 250000 строк, 1 запр/20 сек) — источник: сторонняя новостная сводка. Другой независимый
> аудит того же дня подтвердил `/api/v1/supplier/stocks` как рабочий без замечаний. Ни то ни другое
> не подтверждено по первоисточнику (dev.wildberries.ru блокирует автоматическую проверку) — проверь
> вживую перед использованием.

Лимит ответа: **60000 строк**. Если товаров больше — нужны несколько запросов с offset:

```
GET /api/v1/supplier/stocks?offset=0&limit=60000
GET /api/v1/supplier/stocks?offset=60000&limit=60000
```

**Важно:** История остатков не хранится — только текущие. Обновляются каждые 30 минут.

## Рекомендации

- Всегда использовать **максимальный limit** чтобы минимизировать число запросов (особенно при строгих rate limits)
- Для `reportDetailByPeriod` — **100000**
- Для Content cards — **100**
- Для stocks — **60000**
- Для feedbacks/questions — **50-100**
- Между запросами с пагинацией — соблюдать rate limit конкретного семейства
